## 1. Research Summary

Framework-by-framework findings. Every claim is cited. Sources fetched 2026-04-18.

### 1.1 Claude Code — memory architecture

Source: https://code.claude.com/docs/en/memory

Two orthogonal memory layers:

**CLAUDE.md files** (human-written): hierarchical, loaded by walking the directory tree
upward from cwd. Path-scoped rules in `.claude/rules/*.md` load on-demand when matching
files are opened. Size target: under 200 lines per file — longer files reduce adherence
because they consume more context window.

**Auto memory** (Claude-written): stored at `~/.claude/projects/<project>/memory/MEMORY.md`
with per-topic detail files. The **first 200 lines (or 25KB)** of `MEMORY.md` are loaded
at every session start. Detail files (`debugging.md`, `api-conventions.md`, …) are loaded
**on demand** when Claude accesses related files.

Key design insight: split memory into a **hot index** (always loaded, token-bounded) and
**cold detail files** (loaded only when relevant context activates them). This is a
structural separation, not a retrieval scoring algorithm.

**Absorbable:** the hot-index/cold-detail split maps to emerge's tier system. The missing
piece in M5 is a structured index row (always loaded, small, contains item IDs + brief
summaries) while full item content is fetched lazily via `Memory.get(id)`.

**Anti-pattern:** loading all memory items at session start. Claude Code's 200-line limit
is a structural budget; emerge must enforce it via `RecallBudget.maxTokens` on the
working tier.

### 1.2 Aider — SQLite diskcache for repo map

Sources: https://github.com/Aider-AI/aider (repomap.py), https://aider.chat/docs/config/options.html

Aider uses `diskcache.Cache` (SQLite-backed) at `.aider.tags.cache.v{N}/` in the repo
root. The **version number in the directory name** enables safe cache invalidation
without migration SQL. Each cache entry stores `{mtime, data}`; falls back to in-memory
dict on SQLite failure.

Chat history is Markdown. Issue #3607 (March 2025) reveals the core problem: the only
built-in history control is `/clear` (drops everything). Summarization truncates
important early context while keeping verbose recent chatter — the symptom of no tiered
compression.

**Absorbable:** version-suffix directories for schema invalidation → use SQLite's
`PRAGMA user_version = 1`. On startup, check the version; if it doesn't match, rebuild
(not migrate). Correct for v0.2 where no users have existing SQLite sessions.

### 1.3 Cline — VSCode extension persistence

Source: https://github.com/cline/cline

Cline persists through VSCode's `ExtensionContext.workspaceState` and `globalState` APIs
(JSON, opaque path). Task history also written to `~/.cline/tasks/` as JSON files.
VSCode-API lock-in means the state is inaccessible from a CLI or another editor.

**Absorbable:** one-JSON-per-task under `~/.cline/tasks/` → emerge formalizes as
one-JSONL-per-session under `~/.emerge/sessions/`. JSONL is already `makeRecorder()`
output; M4 makes it the source of truth for resume (not just telemetry).

**Anti-pattern:** tool-specific storage APIs. All emerge persistence goes to filesystem
paths (`~/.emerge/`), process-boundary agnostic.

### 1.4 Continue.dev — LanceDB + SQLite hybrid

Source: https://github.com/continuedev/continue (LanceDbIndex.ts)

Dual-database document index:
- **LanceDB** stores `{uuid, path, cachekey, vector[], startLine, endLine, contents}`
- **SQLite** (`lance_db_cache`) mirrors the same data as durable source of truth

Query pattern: embed query → LanceDB vector search → retrieve full content from SQLite by
uuid. If the LanceDB index corrupts, it rebuilds from SQLite without re-fetching source.

**Absorbable:** source-of-truth / rebuildable-index pattern. M5: SQLite = `memory_items`
(full content, structured attributes); sqlite-vec = rebuildable embedding index.

**Anti-pattern:** storing full content in the vector index. Mixing heavy text with
embedding vectors bloats the file. Keep content in SQLite; keep only `(item_id,
embedding[])` in sqlite-vec.

### 1.5 OpenAI Codex CLI — session model

Source: https://github.com/openai/codex (README; `session-persister.ts` path returned
404 — likely changed in recent refactor) [NEEDS VERIFICATION — source path could not be
confirmed]

From public README: sessions stored as flat JSON arrays per conversation,
UUID-identified, with a separate index file mapping IDs to metadata (start time, title).
No SQL, no vector index. Intentionally minimal; insufficient for cross-session recall.

**Absorbable:** session-index-as-separate-file pattern. M4: maintain

contractRef, schemaVersion}`) so `emerge status` can list sessions without parsing every
JSONL.

### 1.6 OpenDev (arXiv 2603.05344) — compound AI architecture

Source: https://arxiv.org/html/2603.05344

Rust CLI coding agent with a published compound-AI architecture. Four directly absorbable
patterns:

**Dual-agent planner/executor with schema-filtered tools:** Planner's tool schema
includes only read-only tools ("read-only tools excluded from schema entirely"). After
user approval, Executor runs with full tool access. This matches emerge's
`AgentSpec.tools` + `AgentSpec.mode`; needs a `buildPlannerSpec(fullSpec)` convenience
helper.

**Event-driven system reminders:** injected into the **user message role** (not static
system prompt) at decision points. Triggers: iteration count threshold, token pressure,
repeated tool errors, idle detection. Placing reminders in user message role
"strengthens their effect" vs. system prompt. Maps to leaderboard absorption proposal #6
(M4 slot).

**Session persistence:** JSON with atomic writes, session index for fast lookup, cost
metadata, git-based operation logs for rollback. Auto-save prevents data loss on crash.

**5-stage adaptive context compaction:** triggered when token budget approaches
exhaustion. Per-tool-type summarization → large outputs offloaded to external storage
with references → selective compression of older observations. This is the concrete
algorithm for M5's compression pipeline.

**Cross-session playbook:** project-specific knowledge accumulates across sessions as
learned patterns — exactly emerge's `ExperienceLibrary`, validating the M3c2.5
architecture from a published independent source.

### 1.7 AutoGen (v0.4 / AG2) — group chat and debate

Sources: https://github.com/microsoft/autogen,
https://gurusup.com/blog/best-multi-agent-frameworks-2026

`SelectorGroupChat` implements multi-agent debate via 3-stage speaker selection: (1)
custom `selector_func` override → (2) candidate filtering (can exclude previous speaker)
→ (3) LLM-based selection from formatted roster + budget-limited history. Falls back to
previous speaker after `max_selector_attempts` failures.

State persistence keyed by **agent name** (not runtime UUID) for "portability across
different teams." This is the same insight as ADR 0038's `contractId` key for
experiences.

**Microsoft unification (Oct 2025):** AutoGen + Semantic Kernel → Microsoft Agent
Framework (GA Q1 2026). Community fork AG2 continues the original architecture. Design
references below use AG2/original patterns.

**Absorbable:** LLM-based speaker selector for `debate()` topology. The moderator
receives `(roles: string[], truncatedHistory: string)` and returns the next speaker
name. Maps to a `DebateModerator` agent spec with a `select_next_speaker` tool.

**Anti-pattern:** full growing history to moderator on every turn. A 4-agent debate × 5
rounds = 20+ moderator LLM calls, each growing. Apply `RecallBudget` to history slice
(last N turns only) before passing to moderator.

### 1.8 MetaGPT — environment-mediated team coordination

Source: https://github.com/geekan/MetaGPT (team.py)

Pub/sub `Environment` object owns the message hub. Roles subscribe to message types;
team publishes a goal; roles react autonomously. `Team.serialize()` / `deserialize()`
for state persistence. Budget tracking via `cost_manager` with `max_budget` ceiling.

Execution model: `await env.run()` cycles until all roles are idle — **quiescence
detection** as the convergence criterion.

**Absorbable:** the `is_idle` per-role flag for mesh topology termination. When all
bus-observed agents transition to `idle` state, the mesh terminates. Already observable
via `TuiState.agents` (`AgentNode.state`); the `mesh()` builder needs a lifecycle event
subscription.

### 1.9 CrewAI — Flows (deterministic) + Crews (autonomous)

Sources: https://crewai.com/crewai-flows, https://docs.crewai.com/en/introduction

CrewAI 2025 separates concerns: **Flows** (event-driven state machine, `@listen` /
`@start` decorators, 12M+ executions/day) + **Crews** (autonomous role-based teams with
a2a delegation).

Known weakness in hierarchical Crews: the manager executes all tasks sequentially
instead of selectively delegating, inflating latency and token cost. The community fix
is a custom manager with explicit conditional logic.

**Absorbable:** the Flows / Crews two-layer separation validates emerge's
topology-builder (deterministic routing) + agent-runner (autonomous execution)
architecture. No design change needed; useful as documentation framing.

**Anti-pattern confirmed:** sequential task execution in a hierarchical manager.
emerge's `supervisorWorker` with `dispatch: "parallel"` and LLM aggregation (M3c1) is
the correct avoidance.

### 1.10 LangGraph — SQLite checkpointing and structured debate

Sources: https://github.com/langchain-ai/langgraph (SQLite checkpointer),
https://arxiv.org/html/2603.27404v1 (debate via LangGraph),
https://multiagents.org/2025_talks/talk_improving_multi_agent_debate_with_sparse_communication_topology.pdf

LangGraph SQLite checkpointer schema:
```sql
-- Thread-keyed checkpoint blobs
CREATE TABLE checkpoints (
  thread_id            TEXT NOT NULL,
  checkpoint_ns        TEXT NOT NULL DEFAULT '',  -- sub-thread namespacing
  checkpoint_id        TEXT NOT NULL,
  parent_checkpoint_id TEXT,
  type                 TEXT,
  checkpoint           BLOB,                       -- JsonPlusSerializer-serialized state
  metadata             BLOB,
  PRIMARY KEY (thread_id, checkpoint_ns, checkpoint_id)
);

-- Granular channel writes for parallel agents
CREATE TABLE writes (
  thread_id     TEXT NOT NULL,
  checkpoint_ns TEXT NOT NULL DEFAULT '',
  checkpoint_id TEXT NOT NULL,
  task_id       TEXT NOT NULL,
  idx           INTEGER NOT NULL,
  channel       TEXT NOT NULL,
  type          TEXT,
  value         BLOB,
  PRIMARY KEY (thread_id, checkpoint_ns, checkpoint_id, task_id, idx)
);
```

Channel versions: monotonic counter + random suffix `"{v:032}.{h:016}"` — deterministic
ordering without collision.

**Critical difference from ADR 0005:** LangGraph checkpoints between graph nodes.
emerge checkpoints at tool calls. This is the right decision: a long-running tool inside
a LangGraph node loses internal progress on crash — the trap LangGraph users hit
repeatedly in production.

**LangGraph for structured debate** (arXiv:2603.27404): LangGraph's cyclic state machine
is used as the backbone for a multi-phase adversarial debate engine. The state machine
"maintains consistent flow, chat history, and turn logic — preventing chaotic
interference between agents." This validates using a state-machine loop (emerge's
agent runner loop) rather than ad-hoc message passing for the debate topology.

**Absorbable:** the `(thread_id, checkpoint_ns, checkpoint_id)` key triple maps to
`(session_id, agent_role, tool_call_id)` in emerge's schema. The `writes` table provides
sub-checkpoint granularity for parallel agents within a single session.

### 1.11 sqlite-vec — pure-C vector search in SQLite

Sources: https://github.com/asg017/sqlite-vec,
https://alexgarcia.xyz/blog/2024/sqlite-vec-stable-release/index.html

sqlite-vec: pure C, MIT/Apache-2.0, zero native deps, runs on WASM/Raspberry Pi.
Brute-force exact search only (no HNSW in v0.1; roadmap item).

**Benchmark** (100k vectors, 384 dims, float32): ~68ms query, ~1179ms insert. Scale
target: thousands to hundreds of thousands of vectors.

**sqlite-vector** (SQLite AI): SIMD-accelerated alternative
(https://www.sqlite.ai/sqlite-vector). With quantization + preload: ~4ms for same
benchmark (17× faster), perfect recall (Recall@20 = 1.0). Less community tooling; note
for future migration if M5 deployments exceed 1M items.

**For v0.2:** sqlite-vec is the default. 68ms brute-force is acceptable because recall
runs out-of-band (never on the hot path per the `Memory` contract). sqlite-vector is the
documented upgrade path.

### 1.12 LanceDB — hybrid vector + BM25 in TypeScript

Source: https://lancedb.com/blog/openclaw-lancedb-memory-layer

Rust-backed, NAPI TypeScript bindings, Lance columnar format v2.2 (50%+ smaller than
Parquet, 68× faster blob reads). Incremental append writes match agent memory (each
session appends). Hybrid search: vector (IVF-PQ) + BM25 full-text + structured SQL
filtering in one engine.

Production pattern (Continue.dev, OpenClaw, Cognee): LanceDB for vector index + SQLite
for source-of-truth + rebuild LanceDB from SQLite when index corrupts.

**For v0.2:** sqlite-vec (simpler, single-file) is the default. LanceDB is the
documented migration path for deployments needing BM25 hybrid search or exceeding ~1M
items. The `Memory` contract is unchanged; only the implementation swaps.

### 1.13 libSQL / Turso — SQLite fork with edge replication

Source: https://docs.turso.tech/libsql

libSQL is an open-contribution SQLite fork. The `@libsql/client` TypeScript client
supports embedded replicas syncing to a remote Turso database. WAL-frame based sync;
local replica becomes byte-for-byte identical to remote. The full Rust rewrite (Turso
Database, beta in 2026) adds concurrent writes.

**For v0.2:** not needed. emerge v0.2 is self-hosted, single-process, no edge
replication requirement. `better-sqlite3` (synchronous, zero-network) is the right
choice. libSQL/Turso is the future migration path for a "team-shared session store"
feature.

### 1.14 DuckDB — analytical SQL over session logs

Source: https://motherduck.com/learn/duckdb-vs-sqlite-databases/

DuckDB: columnar in-process database, 10-100× faster than SQLite for aggregation
queries. Queries JSONL/Parquet files directly without import.

**For v0.2:** DuckDB is a read-side analytics tool, not a hot-path write store. The
future `emerge analyze [session-glob]` CLI subcommand (v0.3 candidate) will use DuckDB
for cost aggregation, verdict histograms, and experience analysis over session JSONL

#### 1.15.1 ForgeCode (rank 1 + 2) — pre-dispatch correction layer + verification gate

Sources: https://forgecode.dev/blog/gpt-5-4-agent-improvements/,
https://medium.com/@richardhightower/forgecode-dominating-terminal-bench-2-0-harness-engineering-beat-claude-code-codex-gemini-etc-eb5df74a3fa4,
https://github.com/tailcallhq/forgecode

ForgeCode now holds ranks 1 and 2 with the same 81.8% score across two different
backbones (GPT-5.4 and Claude Opus 4.6) — striking confirmation that harness quality
dominates model choice at the top of the board. The four documented harness changes
that closed the GPT-5.4 ↔ Opus 4.6 gap:

1. **JSON schema field reordering** — `required` listed before `properties` to reduce
   GPT-5.4 tool-call errors. (M3b absorbed the shape rewrite; this is the ordering
   detail.)
2. **Nested schema flattening.**
3. **Explicit truncation reminders** when files are partially read.
4. **Mandatory verification pass** — a reviewer skill checks task completion before the
   agent can stop. "Normal prompting — 'please verify your work' — did not produce this
   effect. Enforcement did."

emerge already has #1 + #2 (ADR 0031), #3 (ADR 0033), and #4 (ADR 0035). The new
absorbable pattern is the **community skepticism datapoint** ("benchmaxxed"): top-of-leaderboard
scores from a vendor running its own benchmark are intrinsically suspect. emerge's
v0.2 efficiency-measurement plan (success_rate / token_cost vs. single-agent baseline)
is the credible alternative to chasing absolute scores.

**Absorbable for v0.2:** the explicit field-ordering + flattening discipline informs
how `@lwrf42/emerge-memory-sqlite`'s `attributes_json` schema documentation should
present examples — required fields first, no nested unions in the documented examples.

#### 1.15.2 TongAgents (rank 3) — Planner / Executor / Verifier with adaptive replanning

Sources: https://www.bigai.ai/blog/news/产品能力领跑全球！通研院-tongagents-登上多项国际智能体/,
https://github.com/TongAgents (org page; no public repos as of 2026-04-18 [NEEDS
VERIFICATION — closed source])

TongAgents is BIGAI's agent framework. Architecture is published only in BIGAI's blog
(not on arXiv). Three documented mechanisms:

1. **Adaptive-replanning Planner** — "real-time feedback adjustment capabilities" that
   dynamically adjust plans based on Executor reports, claimed to prevent "getting lost
   in context after dozens of operations."
2. **Three-phase Executor lifecycle** — every subtask runs `execute → report → Q&A`,
   with parallel tool calls, **streaming segmented results for long-running tools**, and
   **async completion notifications**. Explicitly designed for the failure modes
   (timeouts, multi-stage commands) that bite TB 2.0's 100+ task timeouts.
3. **Independent black-box Verifier** — "does not rely on execution history, finding
   potential problems from multiple angles." This is the same insight as KIRA's
   3-perspective `task_complete` verifier (covered in M3b research).

**Absorbable for v0.2 (3 patterns):**

- **The `report` half of `execute → report → Q&A`** — when a `mesh()` peer or
  supervisor-worker child completes, it emits a structured report envelope on the bus
  (not just a result), so the parent can `Q&A` (request clarification) without spinning
  up another full agent invocation. This is a cheap addition to the existing
  `request` / `response` envelope kinds.
- **Streaming segmented results for long-running tool calls** — the existing
  `truncation-aware tool results` (ADR 0033) handles output-after-the-fact; TongAgents'
  pattern handles output-during-the-call. M4 absorbable: emit `tool_progress` events on
  the bus for tools that opt into streaming, recorded in JSONL but not stored as
  separate checkpoints.
- **Async completion notifications for backgrounded commands** — pairs with Droid's
  background-execution primitive (1.15.4) to make the same surface usable on emerge's
  bus. M4 absorbable: a `tool.async_complete` envelope kind that the originating agent
  subscribes to.

description only. Treat as design inspiration, not as patterns observable in shipped

#### 1.15.3 SageAgent / OpenSage (rank 4) — self-generating topology + hierarchical memory

Sources: https://arxiv.org/abs/2602.16891, https://rdi.berkeley.edu/blog/opensage/,
https://github.com/ianblenke/sageagent (community OSS implementation)

OpenSage is the published academic counterpart to TongAgents — UC Berkeley + Google
DeepMind, March 2026. Three pillars (M3b absorption already covered them at a high
level; this update adds the new published numbers):

1. **Self-generated agent topology** — vertical (parent creates sub-agents) + horizontal
   (peer agents in ensemble). Ablation: removing vertical drops 78% → 65% with
   summarization events climbing from 6.4 to 13.1 per task (context overflow forces
   compaction). **Disabled for the TB 2.0 submission** because of TB resource
   constraints; the 78.4% score is from a single-agent variant.
2. **Dynamic tooling system** — agents author task-specific Python / C / C++ tools at
   runtime; on a 300-instance subset, 39 such tools were created. Also disabled for the
   TB 2.0 submission.
3. **Hierarchical graph-based memory** — embedding-based + graph-based retrieval, with
   tailored node/edge types for coding tasks. Ablation: 59.0% on SWE-Bench Pro vs. 56.4%
   for Mem0g vs. 56.2% for NoMem.

**Heterogeneous-model collaboration on TB 2.0** — Gemini 3 Pro for planning + review,
GPT-5 Mini for execution. Concrete instance of mixed-model topologies.

**Absorbable for v0.2 (3 patterns):**

- **Memory-strategy ablation matrix as a deliverable.** OpenSage's published
  `Mem0g vs. NoMem vs. OpenSage memory` ablation is the kind of evidence emerge needs
  for ADR 0040. M5 acceptance: include a `examples/memory-ablation/` demo that runs the
  same task with `none / structural-only / structural+semantic / all-four-strategies`
  and reports the difference, so the multi-strategy claim is backed by numbers.
- **Heterogeneous-model topology** — the `tree()` topology builder should accept a
  per-level model spec (planner level: strong model; executor leaves: weaker / cheaper
  model). No contract change — `AgentSpec.providerHint` already exists. Document this
  pattern as the canonical use of `tree()` in §7.3.
- **Graph-typed memory edges** — emerge's `memory_links` schema (§7.2) needs `kind`
  values that match OpenSage's coding-task taxonomy (`caused | refers | summarizes |
  supersedes | contradicts`), not arbitrary strings. Lock the enum in ADR 0040.

#### 1.15.4 Droid / Factory.ai (rank 6) — sub-agent specialization + background execution

Sources: https://factory.ai/news/terminal-bench, https://github.com/Factory-AI/factory

Droid's published TB 2.0 architecture (M3b research covered an older snapshot; updated
here with the 2026 details):

1. **Environmental context bootstrapping** — every new session bootstraps with broad
   salient system info (languages installed, git repo summary, env vars, running
   processes), presented as **shell command output** (not as a JSON system block) so
   the agent does not redundantly reissue the same commands.
2. **Speed tuning** for TB 2.0's aggressive timeouts.
3. **Background-execution primitive** — controlled, opt-in via settings, dangerous /
   resource-intensive commands filtered, every spawned process tracked for cleanup.
   "Many real workflows … require starting services that outlive the Droid process."
4. **Specialized sub-agent system** — Code Droid / Knowledge Droid / Reliability Droid
   / Product Droid; "Droid is not one agent, it is a system of specialized 'Droids'."
5. **Model-agnostic design** — top scores across Opus, Sonnet, GPT-5; "agent design,
   not just choice of model, is the decisive factor." Same evidence as ForgeCode.

**Absorbable for v0.2 (3 patterns):**

- **Environmental context bootstrap as a built-in tool.** `@lwrf42/emerge-tools` adds
  an opt-in `env.bootstrap` tool that emits `git status`, `ls -la`, `which python|node|
  rustc`, `env` (filtered), `ps aux | head` as a single rich tool result on session
  start. Recorded as a normal tool call so it appears in JSONL / replay. Saves the
  cost of the agent issuing 5-7 individual probes.
- **Background-execution primitive** — a `bash.background(cmd, label)` tool that
  returns a process handle; later `bash.poll(label)` and `bash.kill(label)` tools.
  Maps directly to TongAgents' `tool.async_complete` bus envelope (1.15.2). Process
  handles tracked in the session and surfaced in `WorkspaceManager` for cleanup.
  Required for any TB 2.0 task that runs a server.
- **Specialized sub-agent system as a `BlueprintRegistry` convention** — instead of a
  single "agent" blueprint, ship a small registry of `code-worker / knowledge-worker /
  ops-worker / qa-worker` blueprints with fixed mode + fixed tool restrictions. The
  `tree()` builder (§7.3) consumes them by name, not by raw `AgentSpec`. Improves
  reuse across topology demos.

#### 1.15.5 Capy / capy.ai (rank 7) — parallel agents in isolated VMs with Captain planner

Sources: https://capy.ai, https://capy.ai/articles/aider-vs-capy

Capy is a hosted parallel-agent product (closed source). Architecture documented on
the marketing site:

1. **Captain Capy planner** — a single "model-agnostic planning agent" that decomposes
   work, suggests structured information gathering before ticket creation, and groups
   chats / branches as tasks rather than tabs.
2. **Per-task isolated Ubuntu VM** — every dispatched agent runs in its own sandboxed
   environment with full runtime access (Docker, multi-language toolchains
   pre-installed). Git worktrees by default to avoid merge conflicts and context
   bleed.
3. **Browser-dashboard orchestration** — central control plane; "ship entire sprints
   in parallel" with up to N concurrent agents.
4. **Automated PR + Review flow** — Capy handles branching, PR creation, and review
   without the user touching git; a dedicated Review Agent catches issues
   automatically.

**Absorbable for v0.2 (3 patterns):**

- **Captain planner = `tree()` root supervisor with `dispatch: 'parallel'`** — Capy's
  product-level decomposition validates the design choice already in §7.3. Frame the
  `tree()` demo (`examples/tree-feature-impl/`) explicitly as "the open-source
  primitive for what Capy ships as a product." Concrete framing, not a contract
  change.
- **Git worktree per dispatched agent** — `@lwrf42/emerge-workspaces-git-worktree`
  already supports this; the v0.2 `mesh()` and `tree()` topology builders should
  default to allocating one worktree per leaf agent (configurable). Currently the
  topology builders share a workspace by default.
- **Dedicated Review Agent as a topology member, not a kernel role** — in `tree()`
  with `dispatch: 'parallel'`, declare a `reviewer` peer at the same level as the
  workers; reviewer subscribes to the workers' result envelopes and emits an
  Adjudicator-style verdict before the supervisor aggregates. This composes cleanly
  with the Adjudicator contract (M0) without making the reviewer a kernel-special
  role.

#### 1.15.7 Cross-framework synthesis

The top 7 entries on the TB 2.0 leaderboard collectively show:

- **Harness-engineering > model choice for non-frontier models.** ForgeCode + GPT-5.4 =
  ForgeCode + Opus 4.6 = 81.8%; bare Opus 4.6 ≈ 65%. The same pattern at ranks 6 / 7 /
  9 / 10. emerge's bet on a high-quality harness layer is well-positioned by this data.
- **Verification gates are universal.** ForgeCode (mandatory verifier skill), KIRA

- **Background execution + async notification is the next-most-universal need** after
  verification. Droid ships it; TongAgents ships it; Capy ships it via VM-level

- **Score skepticism is healthy.** ForgeCode's #1 + #2 ranks are on its own benchmark;

  leaderboard score yet" — current README) is honest by comparison; the v0.2 efficiency
  measurement (success_rate / token_cost vs. baseline) is the credible alternative.

### 1.17 Mesh topology — production constraints

Sources: https://gurusup.com/blog/agent-orchestration-patterns,
https://www.augmentcode.com/guides/multi-agent-ai-architecture-patterns-enterprise

## 3. Anti-patterns to avoid (cited)

- **Node-level checkpointing (LangGraph production trap):** ADR 0005 is correct; do not
  regress. Tools are the checkpoint boundary, not graph nodes or agent loop turns.

- **Full content in the vector index (Continue.dev lesson):** keep only `(item_id,
  embedding[])` in sqlite-vec; keep full content in SQLite.

- **Unlimited history without compression (Aider lesson):** `/clear` is a blunt
  instrument; M5 compression must run out-of-band, never blocking the recall hot path.

- **Full growing history to debate moderator (AutoGen cost trap):** apply `RecallBudget`
  to history before passing to the moderator (last N turns only).

- **Agent UUID as checkpoint key:** UUIDs change between sessions. Key checkpoints on
  `(session_id, agent_role, spawn_ordinal)`. Same resolution as ADR 0038's `contractId`
  key.

- **Naive fully-connected mesh beyond 8 agents (N² trap):** enforce `maxMembers`
  ceiling (default 6) in the `mesh()` builder; document that beyond 6, use `tree()`.

- **Debate as default topology (arXiv:2604.02460):** single-agent matches or outperforms
  debate when compute is normalized. `debate()` must be an explicit opt-in, documented
  as expensive. Never auto-selected by surveillance.

- **Pure markdown for structured agent state:** markdown is correct for human-readable
  instructions (Claude Code pattern). emerge's `MemoryItem` has structured `attributes`
  requiring indexed SQL lookups. Pure markdown fails `recall({attributes: {tier:
  "episodic"}})`.

- **VSCode-API-locked persistence (Cline lesson):** all emerge storage goes to
  filesystem paths, process-boundary agnostic.

- **libSQL / Turso for v0.2:** replication is future infrastructure. `better-sqlite3`
  synchronous API matches `SessionRecorder.record()` semantics. Async clients add
  complexity without benefit at v0.2 scale.

- **vis-network for the React dashboard (Langfuse approach):** not React-native;
  requires ref-based DOM mounting and manual state sync. Use `@xyflow/react` instead.

- **Single-vendor benchmark as proof of quality (ForgeCode "benchmaxxed" community
  datapoint):** raw TB 2.0 score on a vendor-controlled benchmark is not credible
  proof. emerge's v0.2 acceptance reports `success_rate / token_cost` vs. single-agent
  baseline as the primary efficiency metric, not absolute leaderboard score.

- **Specialized-sub-agent proliferation without composition (Droid risk):** Droid's
  "Code / Knowledge / Reliability / Product" Droid taxonomy works as a product
  branding choice; for emerge, the same idea must be expressed as composable
  `BlueprintRegistry` entries — adding a new specialist must not require a kernel
  change.

- **Closed-source claims as design source (TongAgents risk):** TongAgents' published
  architecture exists only as marketing copy. Treat absorbable patterns from §1.15.2
  as design inspiration, not as patterns observable in shipped code; mark them
  `[NEEDS VERIFICATION]` in the implementation tickets.

## 4. emerge's distinctive characteristics — what to strengthen in v0.2

emerge is not the first agent harness, the fastest, or the highest-scoring on TB 2.0.
But it occupies a position no other top-10 framework occupies. The four characteristics
below are the project's load-bearing differentiators; v0.2 is a chance to deepen each
one rather than dilute it chasing a leaderboard score.

### 4.1 Frozen contracts since M0 — survived 4 milestone rounds + 2 research absorptions without change

The 31 contract files in `packages/kernel/src/contracts/` have not changed since the
initial commit (M0). Through M1 (kernel runtime), M2 (surveillance), M3a (topologies +
roles), M3b (per-provider schema adapter + verification + truncation), M3c1
(architecture completion), M3c2 / M3c2.5 (CLI + experience library), M3d (TUI +
dashboard), M4-prep (Terminal-Bench runner + Harbor sandbox), and M4 (real-model
validation), no contract has been retroactively modified. Two large research absorption
passes — the 2026-04 Terminal-Bench leaderboard absorption that produced ADRs
0034/0035/0036, and this v0.2 handoff — also did not require a contract change.

Concrete evidence: `examples/types-smoke` instantiates every contract and is part of
CI. The fact that all 30+ packages compile against unchanged contract definitions while
new implementations land at every milestone is the empirical proof.

**No other top-10 TB 2.0 framework publishes this commitment.** ForgeCode iterates its
schema (the Part 2 blog is a frank "we changed four things in the harness"); TongAgents
is closed source; Capy is a hosted product so the contract is whatever the API returns
this week; Droid evolves its Droid taxonomy with each release; SageAgent's framework is
an academic prototype with no contract-stability claim.

**How v0.2 strengthens it:**
- ADRs 0039 / 0040 / 0041 / 0042 must each include a "Contracts touched: NONE" line in
  the consequences section. If a workstream surfaces a need for a contract change, that
  contract change is its own ADR with a separate decision record.
- The `examples/types-smoke` test gains assertions that the new packages
  (`@lwrf42/emerge-persistence-sqlite`, `@lwrf42/emerge-memory-sqlite`,
  `@lwrf42/emerge-experience-sqlite`) satisfy the existing `SessionRecorder` /
  `Replayer` / `Memory` / `ExperienceLibrary` interfaces with no shape extension.
- The handoff makes contract preservation explicit: every "absorbable pattern" in §2 is
  designed to land in a new package or a new function, never as a new method on an
  existing contract.

### 4.2 Verified record-replay reproducibility — real-model evidence

ADR 0023 ("reproducibility, not determinism") established three reproducibility tiers:
record-replay, pinned, free. The record-replay tier was not just designed but verified
end-to-end against a real model in `M4-REAL-VALIDATION-REPORT.md` (Track B, 2026-04-19):
a real `gpt-5.4` session was recorded as JSONL, then replayed with the recorder
disabled, and the replay produced byte-identical envelopes for every tool call. No
other top-10 TB 2.0 framework documents real-model record-replay verification. KIRA's
open source has a deterministic harness but no replay tier; ForgeCode and Droid are
proprietary about their replay infrastructure (if any); SageAgent's paper describes
self-generating topologies that are intrinsically hard to replay.

**How v0.2 strengthens it:**
- M4's `@lwrf42/emerge-persistence-sqlite` extends the JSONL replay surface with SQL-
  indexed checkpoint lookup. The acceptance criterion (§7.1) is explicitly that the
  resumed session produces the same final artifact hash as the original — not just
  "the agent makes progress" but "byte-identical output."
- The `provider_calls` SQLite table (§7.1 schema) records the exact request / response
  pairs so replay can also bypass the provider, not just the sandbox. This is the
  missing piece for replay across long-running tasks.
- A new ADR companion to 0023 is reserved (ADR 0040 referenced in §11) to document
  that semantic recall with embeddings is reproducible-by-recording (the embedding
  query and result are recorded), even though the embedding model itself is
  non-deterministic across runs.

### 4.3 Adjudicator + Custodian + Postmortem self-improving loop — live in `topology-supervisor-worker`

ADR 0029 ("self-improving loop") says: surveillance probes → produces calibration →
informs decomposition → produces sessions → Postmortem analyzes → writes to
ExperienceLibrary → surveillance reads back as priors. As of M3c2.5, this loop is
**live in a working demo**: `examples/topology-supervisor-worker` runs the same task
twice with a shared `HintCountingLibrary`. Run 1 sees zero hints (library empty); Run
2 sees ≥ 1 hint with results from run 1's merged experience. The demo exits 0
end-to-end.

The closest published competitor is OpenDev's "playbook of learned strategies"
(arXiv:2603.05344, March 2026) — but OpenDev's playbook is described, not demonstrated
with a public demo that an outsider can run. emerge's is.

**How v0.2 strengthens it:**
- M5 replaces `InMemoryExperienceLibrary` with `SqliteExperienceLibrary`; the same
  `topology-supervisor-worker` demo gains a "run 1 → kill process → restart → run 2
  still sees hints" assertion (§7.2 acceptance). The self-improving loop now survives
  process restarts, not just within-session memory.
- The new mesh / tree / debate topologies (§7.3) each become demos that participate in
  the same loop: a run leaves a postmortem-distilled experience; subsequent runs in
  the same topology read the hints. Concrete evidence that the loop is general, not
  tied to supervisor / worker.
- The Adjudicator is the verification gate that the rest of the leaderboard converged
  on (Capy Review Agent, Droid QA Droid, KIRA 3-perspective verifier, ForgeCode
  reviewer skill, TongAgents black-box Verifier). v0.2 adds a small ADR clarification
  that emerge's Adjudicator is the named contract for this universal pattern, not a
  bespoke role per topology.

### 4.4 Honest dual thesis — auditability + reproducibility + self-host

The README and current Strategic Pause note state plainly: "infrastructure shipped,
no leaderboard score yet." emerge does not chase TB 2.0 ranks, does not run a
benchmark whose tasks it controls, does not bundle a hosted control plane. The dual
thesis (auditability + reproducibility) is for regulated industries, security teams,
and self-hosters — not the same audience as Capy, ForgeCode, or TongAgents.

This is a positioning differentiator, not a code differentiator — but it is the one
that will determine whether the project finds the three users the strategic pause
identified as a goal.

**How v0.2 strengthens it:**
- The CHANGELOG entry for v0.2.0 must repeat the honest framing: this is
  infrastructure, not a leaderboard play. The new topology demos report
  efficiency-vs-baseline numbers (success_rate / token_cost), not absolute TB scores.
- ADR 0039 explicitly documents the "no encryption at rest" decision so the first
  regulated-industry adopter can challenge it on day 1 instead of finding it as a
  surprise late. Honest-by-design beats benchmarked-by-design.
- The `emerge status` CLI subcommand (M4 deliverable) makes the audit trail visible
  in one command: every session, every cost, every verdict, deep-linkable into the
  dashboard via `?agent=<id>` (§7.4). This is the auditability story turned into a
  shipping artifact.

## 6. Recommended approach per workstream

One concrete recommendation per workstream. Alternatives sketched and rejected so the
chosen approach is auditable.

### 6.1 M4 Persistence — recommended: SQLite (`better-sqlite3`) + JSONL index file

**Pick:** `better-sqlite3` for structured checkpoint and session metadata. Existing
JSONL event stream (ADR 0037) remains the durable event log. The SQLite layer is
additive — it indexes tool-call events for resume queries.

**Justification:**
- JSONL is already the per-event streaming write path. M4 adds a SQLite index over
  tool-call events; it does not replace JSONL.
- ADR 0005: `(session_id, tool_call_id)` is the natural SQLite B-tree primary key.
- `better-sqlite3` is synchronous. `SessionRecorder.record()` is called on the hot
  path (after every event). The synchronous `INSERT` takes ~0.1-0.5ms — well within
  the < 10% overhead budget for any non-trivial tool (100ms+ execution).
- Not libSQL: replication adds infrastructure with no benefit at v0.2 self-hosted
  scale.
- Not DuckDB: columnar format is wrong for hot-path single-record writes.
- Not pure JSONL: no structured queries. Resume needs "which tool_calls in this
  session are already completed for agent_role X?" — a SQL `WHERE`, not a linear scan.

**Package:** `@lwrf42/emerge-persistence-sqlite`

### 6.2 M5 Memory — recommended: SQLite source-of-truth + sqlite-vec semantic index

**Pick:** `MemoryItem` records in SQLite; semantic embeddings in sqlite-vec virtual
table loaded into the same SQLite file; structural / temporal / causal recall via SQL
on the same connection.

**Justification:**
- Three of four `RecallQuery` strategy components (structural, temporal, causal) are
  SQL operations. Only semantic requires a vector index. Keeping all four in one file
  avoids dual-database complexity for the expected v0.2 deployment scale.
- sqlite-vec: MIT/Apache-2.0, pure C, zero native deps, WASM-compatible. 68ms
  brute-force for 100k × 384-dim is acceptable because recall runs out-of-band per the
  `Memory` contract.
- Continue.dev's LanceDB + SQLite dual-DB is correct at 1M+ records or when BM25
  hybrid search is required. Document LanceDB as the migration path in ADR 0040. The
  `Memory` contract is unchanged; only the implementation swaps.
- Embedding runs in a worker thread (`@xenova/transformers`, all-MiniLM-L6-v2, 384
  dims). `embedding_status = 'pending'` queue means the worker never blocks the event
  loop.

**Package:** `@lwrf42/emerge-memory-sqlite`

### 6.3 Topologies — recommended: bus-ACL mesh + nested-supervisor tree + LLM-moderator debate

**Pick:** three new topology builders in `packages/agents/src/topologies/`:

**`mesh()`:** all-to-all bus ACL edges, shared broadcast topic, quiescence termination
(all agents idle per MetaGPT pattern), `maxTurns` ceiling (default 20), `maxMembers`
ceiling (default 6). Pattern: MetaGPT quiescence + LangGraph edgeless routing adapted
to emerge's bus.

**`tree()`:** recursive `supervisorWorker` instances using the existing
`Topology.nested` field. Root supervisor decomposes → mid-level supervisors → leaf
workers. Result propagates upward. The existing `supervisorWorker` is the leaf node;
no contract changes. Per-level `providerHint` accepted (SageAgent heterogeneous-model
pattern).

**`debate()`:** N debaters + 1 LLM moderator. Moderator receives budget-limited
history and calls `select_next_speaker`. Termination: moderator calls `terminate` tool
OR `maxRounds` reached (default 5). Pattern: AutoGen SelectorGroupChat adapted to
emerge's bus.

All three accept a per-leaf workspace strategy (default: one git worktree per leaf
agent — Capy pattern).

### 6.4 Interactive UIs — recommended: `@xyflow/react` for dashboard, panel-switching for TUI

**Pick:**

**Dashboard:** upgrade `TopologyGraph.tsx` from pure SVG to `@xyflow/react`. ELK.js
for auto-layout. Click on agent node → right-panel `AgentInspector`. URL `?agent=<id>`
encodes selected agent. Two new WebSocket RPC frames: `agent.selected` and
`agent.detail`.

**TUI:** add panel-key-switching (`Tab`) cycling topology → verdicts → agent-inspector
→ cost. Arrow keys navigate agents in topology panel. `Enter` selects agent for
inspector. `Escape` returns to topology panel. No graph library — highlighted-border
tree rendering.

**Rationale for `@xyflow/react`:** the only graph library that is React-native AND
supports full HTML DOM nodes (required for status badges and click handlers).
vis-network and Cytoscape.js require ref-based DOM mounting with manual React state
sync. Pure SVG cannot support click events and zoom/pan for mesh cross-edges.

async function executeOrReplay(

  completedCalls: ReadonlyMap<ToolCallId, ToolResult>,
  sandbox: Sandbox,

**RecallTrace implementation sketch (4-strategy):**
```typescript
// packages/memory-sqlite/src/recall.ts

export async function recall(
  db: Database,            // better-sqlite3 connection
  query: RecallQuery,
  scope: RecallScope,
  budget: RecallBudget,
): Promise<RecallResult> {
  const OVER_FETCH = 3;   // over-fetch by 3× then re-rank
  const limit = (budget.maxItems ?? 20) * OVER_FETCH;
  const scores = new Map<MemoryItemId, RecallTraceEntry>();

  // 1. SEMANTIC: sqlite-vec brute-force (out-of-band embedding worker)
  if (query.text) {
    const embedding = await embedQueue.embed(query.text);  // worker thread
    const rows = db.prepare(
      `SELECT item_id, distance FROM memory_embeddings
       WHERE embedding MATCH ? AND k = ?`
    ).all(serializeVec(embedding), limit) as { item_id: string; distance: number }[];
    for (const r of rows) merge(scores, r.item_id as MemoryItemId, { semantic: 1 - r.distance });
  }

  // 2. STRUCTURAL: SQL WHERE on JSON attributes
  if (query.attributes) {
    const whereClauses = Object.entries(query.attributes).map(
      ([k, _]) => `json_extract(attributes_json, '$.${k}') = ?`
    );
    const rows = db.prepare(
      `SELECT id FROM memory_items WHERE ${whereClauses.join(' AND ')}
       AND session_id IN (${scopeSessions(scope)}) LIMIT ?`
    ).all(...Object.values(query.attributes), limit) as { id: string }[];
    for (const r of rows) merge(scores, r.id as MemoryItemId, { structural: 1.0 });
  }

  // 3. TEMPORAL: SQL WHERE on created_at
  if (query.since !== undefined || query.until !== undefined) {
    const rows = db.prepare(
      `SELECT id FROM memory_items
       WHERE session_id IN (${scopeSessions(scope)})
         AND created_at BETWEEN ? AND ? LIMIT ?`
    ).all(query.since ?? 0, query.until ?? Date.now(), limit) as { id: string }[];
    for (const r of rows) merge(scores, r.id as MemoryItemId, { temporal: 1.0 });
  }

  // 4. CAUSAL: recursive CTE on memory_links (typed edges per SageAgent taxonomy)
  if (scores.size > 0) {
    const seeds = [...scores.keys()].map(() => '?').join(',');
    const rows = db.prepare(
      `WITH RECURSIVE causal(id, depth) AS (
         SELECT to_id, 1 FROM memory_links WHERE from_id IN (${seeds})
         UNION ALL
         SELECT ml.to_id, c.depth + 1
         FROM memory_links ml JOIN causal c ON ml.from_id = c.id
         WHERE c.depth < 3
       )
       SELECT DISTINCT id FROM causal`
    ).all(...scores.keys()) as { id: string }[];
    for (const r of rows) merge(scores, r.id as MemoryItemId, { causal: 0.5 });
  }

  // 5. Sort, apply token budget, build RecallTrace
  const ranked = [...scores.values()].sort((a, b) => totalScore(b) - totalScore(a));
  const kept = ranked.slice(0, budget.maxItems ?? 20);
  const items = fetchItems(db, kept.map(e => e.itemId));

  return {
    items,
    trace: { items: kept, droppedForBudget: ranked.length - kept.length },
  };
}
```

**Compression layers:**
```
Working tier compression (sync, triggered in-process):
  Trigger: total tokens in 'working' tier for session > threshold (default 4000 tokens)
  Action:  run a summarize pass via the kernel's weak-model provider config,
           store summary as new 'episodic' item,
           retier originals to 'archived'
  Pinned items: excluded from trigger check (idx_mem_pin)

Episodic → semantic index (async, worker thread):
  Trigger: embedding_status = 'pending' rows (polled every 5s by worker)
  Action:  embed content in batches of 50 via @xenova/transformers,
           insert into memory_embeddings,
           set embedding_status = 'done'
  Never blocks the event loop.

Semantic → archived (time-based expiry, configurable):
  Trigger: item created_at older than N days (default: 30) AND pin IS NULL
  Action:  retier to 'archived'; excluded from default recall scope
           but accessible via explicit Memory.get(id)
```

    edges,
  };

**Real-task efficiency measurement plan:**
Run each topology on a fixed Terminal-Bench-style task at 3 agent counts (3 / 5 / 7
for mesh; 2 / 3 / 4 levels for tree; 2 / 3 / 4 debaters for debate). Measure: (1)
task success rate, (2) total LLM calls, (3) wall-clock time, (4) total token cost.
Report as a 3 × 4 matrix per topology. "Emergent efficiency" = `success_rate /

import { ReactFlow, Background, Controls, MiniMap, useNodesState, useEdgesState } from '@xyflow/react';
import '@xyflow/react/dist/style.css';
import ELK from 'elkjs/lib/elk.bundled.js';

const elk = new ELK();

async function computeLayout(agents: ReadonlyMap<AgentId, AgentNode>): Promise<{nodes: Node[]; edges: Edge[]}> {
  const elkGraph = {
    id: 'root',
    layoutOptions: { 'elk.algorithm': 'layered', 'elk.direction': 'DOWN' },
    children: [...agents.values()].map(a => ({ id: String(a.id), width: 140, height: 48 })),
    edges: [...agents.values()]
      .filter(a => a.parentId !== undefined && agents.has(a.parentId))
      .map(a => ({ id: `${String(a.parentId)}->${String(a.id)}`, sources: [String(a.parentId)], targets: [String(a.id)] })),
  };
  const laid = await elk.layout(elkGraph);
  const nodes: Node[] = (laid.children ?? []).map(n => ({
    id: n.id, position: { x: n.x ?? 0, y: n.y ?? 0 }, data: agents.get(n.id as AgentId)!, type: 'agentNode',
  }));
  const edges: Edge[] = (laid.edges ?? []).map(e => ({ id: e.id, source: e.sources[0], target: e.targets[0] }));
  return { nodes, edges };
}

export function TopologyGraph({ state, onAgentSelect }: Props): React.ReactElement {
  const [nodes, setNodes, onNodesChange] = useNodesState([]);
  const [edges, setEdges, onEdgesChange] = useEdgesState([]);

  useEffect(() => {
    computeLayout(state.agents).then(({ nodes: n, edges: e }) => { setNodes(n); setEdges(e); });
  }, [state.agents]);

  return (
    <ReactFlow
      nodes={nodes} edges={edges}
      onNodesChange={onNodesChange} onEdgesChange={onEdgesChange}
      nodeTypes={{ agentNode: AgentNodeComponent }}
      onNodeClick={(_, node) => onAgentSelect(node.id as AgentId)}
      fitView
    >
      <Background /><Controls /><MiniMap />
    </ReactFlow>
  );
}

## 8. Cross-cutting concerns

**Test strategy:**
- M4: `examples/resume-demo/` with real-model run + kill + restart + verify. Uses
  `EMERGE_LLM_*` env vars; skips cleanly if unset. Success criterion: identical
  final artifact hash for `fs.write` tools; bash tool results allowed to differ
  (timestamps etc.).
- M5: `examples/memory-scale-demo/` with 100-run loop (mock provider acceptable for
  most runs; local embedding model via `@xenova/transformers`). Verifies: hints
  grow, budget respected, pinned items never dropped. Plus
  `examples/memory-ablation/` for the 4-strategy ablation matrix.
- Topologies: one demo per topology (`examples/mesh-bug-triage/`,
  `examples/tree-feature-impl/`, `examples/debate-security-review/`). At least one
  topology must show emergent efficiency (success_rate / token_cost >
  single-agent baseline).
- UI: vitest + jsdom for components; `@xyflow/react` ships its own testing
  utilities. Playwright smoke for click-to-inspect (dev dependency; not required
  for v0.2.0 CI gate).

**Backwards compat:**
- `Memory` contract frozen at M0. `SqliteMemory` is a new implementation;
  `SimpleMemory` continues unchanged.
- `SessionRecorder` contract frozen. `SqliteRecorder` is a new implementation;
  JSONL recorder continues unchanged.
- `Kernel.spawn()` with no `SpawnOptions` behaves identically to v0.1. The
  `SpawnOptions` parameter is optional.
- All published `@lwrf42/emerge-*` packages publish without breaking changes.

**Performance budget (persistence < 10% latency overhead on hot path):**
- `better-sqlite3` synchronous `INSERT`: ~0.1-0.5ms on SSD. A tool call typically
  takes 100ms-30s (sandbox execution). Overhead: < 1% for any non-trivial tool.
  Acceptable.
- Embedding generation (M5): worker thread, never blocks event loop. The
  `embedding_status = 'pending'` queue decouples embedding from recall. New
  items fall back to structural / temporal / causal recall until embedding is
  ready.
- `@xyflow/react`: ~200KB added to dashboard bundle. The dashboard is a dev /
  ops tool, not a production client-facing surface. Acceptable.

**Security: encryption-at-rest decision (ADR 0039 deferral):**
Decision for v0.2: **no encryption at rest.** Rationale: emerge is self-hosted;
SQLite files live in `~/.emerge/` with user-level filesystem permissions.
SQLCipher or SQLite3MultipleCiphers add a native addon dependency and key
management complexity out of scope for v0.2. Mitigation: `emerge init` writes
`~/.emerge/README.md` documenting that the directory contains session data and
recommending filesystem encryption (LUKS, FileVault) for sensitive deployments.
Revisited for v0.3 or when the first regulated-industry adopter requires it.

## 10. Open questions to resolve early (10)

1. **Local embedding model vs. API embedding:** `@xenova/transformers`
   (all-MiniLM-L6-v2, 384 dims, ~40MB download, pure JS / WASM, zero API cost)
   vs. OpenAI text-embedding-3-small (1536 dims, API cost, better quality). The
   local model avoids API dependency but adds a download; 1536 dims with
   sqlite-vec at 100k items is ~1s recall (marginal). Recommendation: default
   to local 384-dim model; make embedding pluggable via an `Embedder`
   interface in `@lwrf42/emerge-memory-sqlite`.

2. **Encryption at rest:** documented above as out-of-scope for v0.2. When does
   this become a blocker? Specifically: will the first regulated-industry
   adopter (the dual thesis's target user) accept unencrypted session data on
   their filesystem?

3. **`agent_role` uniqueness within a session:** in a worker pool of 5
   identical workers, `agent_role` is not unique within a session.
   Resolution: key on `(session_id, agent_role, spawn_ordinal)` where
   `spawn_ordinal` is the Nth agent of that role in the session (tracked by
   `Kernel.spawn()`). Schema in §7.1 already reflects this; confirm before
   writing.

4. **ELK.js in the browser:** ELK's WASM build adds ~300KB to the dashboard
   bundle. Alternative: Dagre (~50KB, hierarchical only — wrong for mesh).
   Decision: use ELK.js for all topologies with non-hierarchical edges (mesh,
   debate); use BFS for simple supervisor-worker trees (existing behavior,
   free). This hybrid minimizes bundle impact.

5. **Debate termination criterion:** three options — (a) moderator explicitly
   calls `terminate` tool (cleanest, but the moderator may not), (b) fixed
   `maxRounds` ceiling, (c) Adjudicator-style verdict on the transcript.
   Recommendation: implement both (a) and (b); (b) is the safety net.
   Document (c) as a future option.

6. **Mesh broadcast semantics:** should all mesh members see all messages on
   the shared topic, or should agents selectively address peers? Fully-shared
   topic is simple but creates noise for large meshes (which are bounded at
   6). Recommendation: fully-shared topic for v0.2 (simpler, bounded scale);
   selective routing is a v0.3 option.

7. **Time-travel in the `ReplayScrubber`:** upgrade from "replay from event
   0" to "jump to checkpoint N then replay forward"? This requires storing
   serialized `TuiState` snapshots at each checkpoint boundary in the SQLite
   `checkpoints` table. Adds complexity to M4. Recommendation: implement
   checkpoint-aware scrubber as a stretch goal in Week 12; core M4 is the
   tool-call resume, not the UI feature.

8. **Package naming for persistence:** `@lwrf42/emerge-persistence-sqlite`
   (recommended) vs. `@lwrf42/emerge-session-store-sqlite` vs.
   `@lwrf42/emerge-store`. The pattern `emerge-experience-inmemory`
   establishes: `emerge-{domain}-{backend}`. So:
   `emerge-persistence-sqlite` for M4, `emerge-memory-sqlite` for M5,
   `emerge-experience-sqlite` for M5 ExperienceLibrary backend.

9. **Resume + Adjudicator:** when resuming a session mid-way, does the
   Adjudicator re-evaluate tool calls that were already completed before the
   crash? The correct answer is no: Adjudicator state (verdicts) should also
   be checkpointed. The `verdicts` table in §7.1 schema reflects this;
   confirm before implementation.

10. **`@xenova/transformers` download on first use:** ~40MB model download
    on the first session that uses semantic recall. User experience
    implication: add a progress bar to the CLI / TUI for the first-time
    download, or pre-download via `emerge init`. Decision needed before M5
    ships.
